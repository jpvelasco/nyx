Why a spec?

Most homelab and lab networks are built with good intentions but rarely verified over time. nyx lets you write down exactly how you expect your network to behave — then continuously checks that reality matches your intent.

The spec is the single source of truth. It is deliberately human-readable and version-controllable.

THE MODEL

Spec Structure

A nyx spec has two layers. The first layer describes how you intend your network to be structured and what should be allowed. The second layer — assertions — contains the actual checks nyx will run to verify that reality matches your intent.

Most fields exist to give context and meaning to your assertions. The more accurately you model your network, the more useful the results become.

Top-level fields

Identity
version
Always 1
site
Human-readable name for this environment
Network Model
networks
The segments/zones in your network (CIDRs, gateways, optional VLANs)
policies
Your declared access rules between zones (allow/deny intent)
vpn
Expected VPN tunnels and the routes that should use them
probes
Optional remote hosts (via SSH) so checks can run from other segments
Executable Checks
assertions
The actual checks nyx will execute. This is the core of the spec.
examples/homelab.yaml (simplified)
version: 1
site: home-lab

networks:
  - name: trusted
    cidr: 10.0.10.0/24
    gateway: 10.0.10.1
    zone: trusted
    vlan: 10
  - name: iot
    cidr: 10.0.60.0/24
    gateway: 10.0.60.1
    zone: iot
    vlan: 60

policies:
  - name: iot-isolation
    from: iot
    to: trusted
    action: deny

assertions:
  - type: subnet_discovery
    network: trusted
    expect_hosts_min: 10
    expect_hosts_max: 30

  - type: isolation
    from: trusted
    to: iot
    expect: deny
THE CHECKS

Assertion Types

These are the live validations nyx can perform.

subnet_discovery
Discovers hosts on a network using nmap and validates host counts.
expect_hosts_min
expect_hosts_max
isolation
Verifies that one zone cannot reach another (or can, if expected).
from, to, expect: deny|allow
route_check
Confirms a specific target is reachable via the expected gateway.
target
vpn_route
Ensures traffic to a target actually flows through the declared VPN tunnel.
vpn, target, expect_tunnel
network_health
Measures latency and packet loss to a target.
target, expect_latency_ms, expect_loss_pct
port_check
Verifies specific TCP ports are open and reachable.
target, ports[], scan_mode
dns_check
Validates that a hostname resolves to the expected IP.
query, expect_ip, server
acl_check
Verifies that a policy is actually enforced by a supported controller (Omada, future OPNsense).
provider, policy, expect: enforced

Advanced Capabilities

Probes & Runners
VANTAGE AWARENESS

Run assertions from inside other VLANs using SSH probes. This is essential for accurate isolation testing of IoT or guest networks.

runner: iot-probe
Providers
Omada — import + live ACL enforcement checks
OPNsense — import + check
Providers register themselves and expose nyx omada and nyx opnsense subcommands.

What nyx Delivers

First Contact
`doctor` and bare invocation instantly tell you where you are on the network relative to your spec, including multi-homed detection and clear next steps.
Recommendations Engine
When things fail, nyx doesn't just say "fail". It classifies root causes (vantage, isolation breach, service down, etc.) and proposes concrete fixes with ready-to-paste YAML patches.
Snapshot + Drift
Save known-good states. Later, `drift status` shows exactly what changed — new failures, fixed issues, or degradations — giving you long-term confidence instead of one-off audits.
Made for homelab and lab operators who want to sleep at night.
